How Does OptimusCloud Work?

Step 1: Configuration Stage

At the configuration stage, the user chooses workload and database-specific settings. From this stage, the user can send the selected configuration to the OptimusCloud Server at any time. There can only be one workload set up in the configuration stage at a time. As soon as the “Run Workload” button is selected, the workload leaves the configuration stage and the OptimusCloud Server receives the workload.


Step 2: Init Stage

During the init stage, the workload is taken in by the OptimusCloud server and begins to be processed. Here, the server initializes and configures all of the VMs. This state is finished once the VMs have been initialized or if the workload gets canceled.


Stage 3: Run Stage

In this stage, the workloads are running with their connection to the OptimusCloud server. This stage ends when the user selects the “Stop Workload” button.


Stage 4: Closing Stage

In the closing stage, the VMs close for the specified workload and additional cleanup is performed. Results are displayed.



Definitions and Descriptions

Workload Configuration

General

  • Workload Distribution: The mix of the four different types (Read, Write, Scan, and Update) of database requests in the chosen workload.
    • Read: A database read request.
    • Write: A database write request.
    • Update: A request to update a record in the database.
    • Scan: A request to scan through all of the entries in a database table.

Volume Configurations

  • Number of nodes: The number of cloud virtual machine instances to run the specified database on.
  • Frequency of change: This is the frequency at which the workload will change.
  • Workload length: This is the amount of time that a workload will be run.


The “upload trace configuration” must be a JSON file, in the same way that the “download current trace” is a JSON file. The format must be the following:

{“state”: state_value, “distribution”: [{“start”: start_value, “end”: end_value, “distribution”: {“SCAN”: value, “UPDATE”: value, “WRITE”: value, “READ”: value}}, {“start”: start_value, “end”: end_value, “distribution”: {“SCAN”: value, “UPDATE”: value, “WRITE”: value, “READ”: value}},], “node_count”: 6}

Essentially, the file must specify a state value, which will generally be 0. Then they must specify the workload distribution, which will have the scan/update/write/read values, separate for each start and end time.
